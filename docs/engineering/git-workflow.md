# Git Workflow

## 브랜치 운영

- 현재는 1인 개발 저장소이므로 `main` 중심으로 운영한다.
- 별도 브랜치 전략은 기본 규칙으로 두지 않는다.
- 큰 작업을 임시로 분리하고 싶을 때만 선택적으로 브랜치를 만든다.

## 커밋

- Conventional Commits를 사용한다.
- 하나의 커밋은 하나의 의도를 가져야 한다.
- 빌드가 깨지는 중간 상태 커밋은 피한다.

## 푸시

- 푸시 전 `npm run build`를 통과시킨다.
- 푸시 전 민감정보 검사 훅을 통과해야 한다.
- 실제 비밀값이 포함된 파일은 푸시하지 않는다.

## 민감정보 관리

- `.env`는 로컬 전용 파일이다.
- 예시 값은 `.env.example`에만 둔다.
- API 키, 액세스 토큰, 개인 비밀번호, private key는 저장소에 넣지 않는다.
