# AGENTS.md

이 저장소에서 작업하는 Codex와 사람 작업자는 아래 규칙을 따른다.

## 목적

- 삼박한최 공방 웹사이트와 관리자 기능을 일관된 방식으로 유지한다.
- 코드 스타일, 커밋 메시지, 푸시 전 안전 점검을 문서와 자동화로 함께 관리한다.

## 작업 원칙

- 변경은 작고 명확하게 유지한다.
- 한 변경은 하나의 의도를 가져야 한다.
- 중복보다 추상화를 서두르지 않는다.
- 의미가 분명한 이름을 우선한다.
- 숨겨진 부작용보다 명시적인 데이터 흐름을 선호한다.
- 실패 가능성이 있는 작업은 사용자에게 보이는 방식으로 처리한다.

## 코드 컨벤션

- 클린 코드 원칙을 따른다.
- 함수는 한 가지 책임만 갖도록 유지한다.
- 컴포넌트는 화면 구조와 데이터 처리 책임이 과도하게 섞이지 않도록 분리한다.
- 이름은 역할이 드러나도록 작성한다.
- 축약어는 널리 알려진 경우가 아니면 피한다.
- 불필요한 주석은 쓰지 않는다.
- 주석이 필요하다면 왜 그런 선택을 했는지 설명한다.
- 매직 넘버와 매직 스트링은 바로 이해되지 않으면 상수로 뺀다.
- 조건문은 단순하게 유지하고 중첩을 줄인다.
- 데이터 타입은 최대한 명확하게 선언한다.
- `any`는 피한다. 정말 필요한 경우에만 이유가 분명해야 한다.
- null/undefined 가능성은 숨기지 말고 코드에서 직접 다룬다.
- UI 문구와 운영 데이터는 하드코딩이 불가피한 경우를 제외하면 한 곳에서 관리한다.
- 새 기능은 기존 디자인 톤과 정보 구조를 해치지 않게 붙인다.

## 프론트엔드 구현 규칙

- 현재 프로젝트는 `Vite + React + TypeScript` 기준으로 유지한다.
- 페이지 컴포넌트는 사용자 경험에 집중하고 데이터 접근은 `src/app/lib` 계층으로 모은다.
- 관리자 기능과 공개 페이지의 데이터 접근 규칙은 가능한 한 동일한 저장소 계층을 사용한다.
- 환경 변수는 `VITE_` 접두사만 사용한다.
- 실제 비밀값은 절대로 코드나 문서에 하드코딩하지 않는다.

## 테스트와 검증

- 이 프로젝트에서는 별도 테스트 코드 작성을 기본 요구사항으로 두지 않는다.
- 대신 변경 후 최소한 아래 검증은 수행한다.
- `npm run build`

## 커밋 메시지 규칙

- Conventional Commits를 사용한다.
- 형식: `<type>: <summary>`
- 필요하면 `type(scope): <summary>` 형식을 사용한다.
- summary는 짧고 명확한 영어 문장으로 작성한다.
- 첫 글자는 소문자로 시작한다.
- 마침표는 붙이지 않는다.

### 허용 타입

- `feat`
- `fix`
- `refactor`
- `style`
- `docs`
- `chore`
- `perf`
- `build`
- `ci`
- `revert`

### 예시

- `feat: add admin inquiry dashboard`
- `fix: handle missing supabase env values`
- `docs: document repository workflow`

## Git 작업 규칙

- 혼자 작업하는 저장소이므로 복잡한 브랜치 전략은 두지 않는다.
- 기본 브랜치는 `main`으로 유지한다.
- 작업 단위가 명확하면 `main`에 직접 커밋하고 푸시할 수 있다.
- 단, 푸시 전 안전 점검을 반드시 통과해야 한다.

## 푸시 전 안전 점검

- 실제 API 키, 토큰, 비밀번호, 비밀값이 포함된 파일은 절대 푸시하지 않는다.
- `.env` 파일은 커밋하지 않는다.
- 예시 값은 `.env.example`에만 둔다.
- 민감정보 점검 훅이 실패하면 원인을 해결하기 전까지 푸시하지 않는다.

## 실행 명령

- 의존성 설치: `npm install`
- 개발 서버: `npm run dev`
- 프로덕션 빌드: `npm run build`

## 변경 전 확인

- 관련 파일 구조를 먼저 읽고 수정 범위를 좁힌다.
- 사용자 변경사항을 임의로 되돌리지 않는다.
- 기존 규칙과 충돌하는 변경이라면 먼저 규칙을 갱신한 뒤 작업한다.
